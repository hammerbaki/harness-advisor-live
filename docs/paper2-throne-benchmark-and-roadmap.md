# Paper 2 — Throne 벤치마킹 분석 및 후속 작업 로드맵

> 상태: 기획 초안 (draft). 논문 2(투자 *정보* 제공 에이전트 + 사용성 평가)를 위한
> 벤치마킹 분석과 작업 로드맵. 논문 1(제출 완료된 harness-engineering 아티팩트)의
> 동결 상태를 훼손하지 않는 작업 공간을 전제로 한다.
> 분석 시점: 2026-07-15 KST, **공개 접근 가능한 범위만** 확인.

## 0. 목표

- **연구 목표:** 투자 *정보*(조언 아님)를 제공하는 에이전트를 만들고 **사용성 평가** 수행.
- **제품 참조점:** [Throne (쓰론)](https://throneinvest.ai/) — 동일 철학의 상업적 성숙 버전.
- **범위 원칙:** Throne의 *데이터 해자*를 복제하지 않는다. 사용성 연구가 성립하는 데
  필요한 만큼만 UX 성숙도를 올린다. 목표는 복제가 아니라 **"연구적으로 검증 가능한
  투자정보 리서치 에이전트"**.

## 2. Throne 역설계 — "챗봇"이 아니라 **리서치 워크플로 제품**

핵심 판단: Throne은 단순 투자 Q&A 챗봇이 아니라
**질문 → 데이터 검색 → 근거 기반 답변 → 투자 노트/문서화 → 공유 문서에서 다시 분석 시작**
으로 이어지는 리서치 워크플로 제품이다.

랜딩 카피: "정답을 찍어주는 대신 판단 기준을 만든다" + 투자권유 아님 강한 고지.
차별점 강조: FnGuide·DART·KOSCOM·KRX·SEC·FMP·Massive를 자연어 질문에 연결하고,
사용자의 리서치 흐름을 기억하며, 결과를 투자 노트로 남긴다.

### 추정 아키텍처 (공개 증거 기반)

| 계층 | 관찰된 것 | 시사점 |
|---|---|---|
| 프런트 | **Next.js** (`x-powered-by: Next.js`, RSC/Next Flight 스트림), PWA manifest, OpenGraph/Twitter/JSON-LD SEO | SEO·공유 최적화된 단일 앱 |
| 인증 | `/api/auth/google` (Google OAuth) | 가입 마찰 최소화 + 무료 체험 |
| 공개 문서 | SEO 가능한 `article` route, sitemap 27개 | 채팅 결과를 버리지 않고 **문서로 제품화** |
| 문서 시스템 | 블록 문서/마크다운/태그/티커/조회수/공유 CTA | 에디터·문서 시스템이 제품 핵심 |
| API(추정) | session, completion, market-data, knowledge-base, public feed/posts, admin KB | RAG/지식베이스 계층 (`bedrock-kb`류 명명) 시사 |
| **데이터(해자)** | FnGuide, KOSCOM, KRX, DART, SEC, FMP, Massive | 상당수 **유료 기관 라이선스** |
| GTM | Telegram 채널, freemium, 공개 문서 SEO 유입 | 콘텐츠·커뮤니티 루프 |

공개 문서 27개 성격: 종목 분석, ETF 전략, 시장 이벤트 해설, 13F/정책 분석,
퀀트 스크리닝, 개인 투자 노트.

### 노출된 데모 질문 예시
- "반도체 업황이 개선되고 있나?" → 수출입·환율·원자재 교차 참조
- "삼성전자 실적이 왜 좋아졌나?" → 재무제표 추세 + 애널리스트 리포트 상관
- "최근 DART 공시에서 투자자가 주목할 변화는?" → 공시 핵심 변화 추출
- "애플, 아직 비싼가?" → 재무 + SEC + 밸류에이션 배수 동시 참조

## 3. 핵심 발견 — 철학의 일치, 그러나 우리가 더 엄격해야 함

Throne이 명시하는 세 가지(비추천·판단보조·출처표시)는 논문 1의 harness가 *코드로 강제*하는
세 가지와 동일하다. Throne은 우리 포지셔닝의 상업적 검증 사례다.

**단, 주의:** Throne의 공개 문서 중 일부는 전략/비중/매수 트리거에 가까운 표현을 포함한다.
논문·사용성 평가용 제품이라면 오히려 **현재 리포의 추천어 차단과 출처 계약을 더 엄격하게
유지**하는 편이 안전하다. 이것이 우리의 차별점이자 방어선이다.

## 4. 벤치마킹 갭 분석

| 축 | 현재 리포 | Throne |
|---|---|---|
| 강점 | 출처 기반 클레임, 검증 시나리오, 가드레일, 재현성, 논문용 평가 자산 | 제품 포장, 데이터 범위, 세션/노트/공유 루프, 랜딩/SEO |
| 안전/포지셔닝 | 코드-소유 harness로 강제 | 프롬프트 기반(추정), 일부 문서는 전략어 근접 |
| 데이터 폭 | 공개(DART/KRX/공개 기업자료), 5개 그룹 | 유료 기관 피드 + 미국 시장 |
| 인터랙션 | 단발 브리핑/Q&A | 대화형 + 메모리 + 노트 + 공유 문서 |
| 배포 성숙도 | fixture/live graceful-degradation | 실서비스(OAuth·freemium·SEO) |

**두 통찰**
1. Throne의 해자는 LLM이 아니라 **데이터 라이선스 + 문서/워크플로 제품화**다. 데이터 폭은
   비용 장벽 → 복제하지 않는다. 문서/워크플로 감각은 저비용으로 벤치마킹 가능.
2. 우리의 신규성은 **코드로 강제된 안전성(추천 차단·출처 계약)** 이다. Throne이 검증 가능한
   형태로 보여주지 못하는 것을 우리는 보유.

## 5. 후속 작업 로드맵

> 경고: "Throne 수준까지 제품 발전"은 *제품 목표*이고 논문 2의 *사용성 평가 연구*와 다르다.
> 데이터 폭·미국 시장을 다 쫓으면 연구 타임라인을 잠식한다.

### 즉시 착수 — 구현 전에 먼저 문서화할 4개 산출물 (초안 작성됨)
1. **경쟁제품 매트릭스** — [`paper2-competitor-matrix.md`](paper2-competitor-matrix.md)
   (Throne 및 유사 서비스 vs 우리 — 축: 안전, 데이터, 워크플로, 배포).
2. **안전·범위 정책** — [`paper2-safety-scope-policy.md`](paper2-safety-scope-policy.md)
   ("출처 기반 투자정보 이해 지원 에이전트"로 범위 고정; 추천어 차단·출처 계약 명문화).
3. **MVP 화면 흐름** — [`paper2-mvp-screen-flow.md`](paper2-mvp-screen-flow.md)
   (질문 → 근거 패널 → 답변 → 저장 노트 → 공유 리서치 문서).
4. **사용성 평가 태스크 5개** — [`paper2-usability-tasks.md`](paper2-usability-tasks.md)
   ("왜 주가가 움직였나", "공시 핵심 변화 찾기", "두 종목 비교", "출처 검증", "리스크 체크리스트").

### A. 제품 범위 고정 (positioning)
- "개인화 투자 조언"이 아니라 **"출처 기반 투자정보 이해 지원 에이전트"**로 정의.
- Throne을 *제품 목표*가 아니라 **상업적 비교 기준선**으로 규정. 연구의 승부처는
  신뢰 보정(trust calibration)·적정 의존(appropriate reliance).

### B. MVP는 채팅이 아니라 **워크플로**로 설계
- 질문 → 근거 패널 → 답변 → 저장된 투자 노트 → 공유 가능한 리서치 문서까지 한 흐름.
- 이렇게 해야 Throne 수준의 "제품감"이 난다. 단발 Q&A로는 부족.
- 제약: `server/index.mjs`의 계약 검증(leakage/link/language 3계열)을 깨지 않을 것.

### C. 데이터 계층은 단계적 확장
- 현재 DART/KRX/공개 기업자료 유지 → 이후 **SEC/FMP/Polygon류 해외·시세 데이터** 부착.
- FnGuide/KOSCOM류 유료 데이터는 라이선스·비용이 선결 → 후순위/스코프 아웃.

### D. UI: 모바일 브리핑 → 데스크톱 리서치 워크스페이스
- 왼쪽 채팅, 오른쪽 출처/표/차트, 하단 또는 별도 패널에 노트/문서 편집.

### E. 평가 설계를 제품 기능과 함께
- 위 태스크 5개로 사용성 평가 구성. formative(n≈5–8) vs summative(n≈20–40+) 결정.
- IRB + 경쟁 서비스 이용약관(실험 사용 가능 여부) 검토.
- 실데이터 라이브 경로 안정화(latency·신선도)로 환경 민감성이 사용성 점수를 오염시키지
  않도록.

### F. 논문 기여 프레이밍
- "벤치마크 제품을 모방했다"가 아니라 **"출처 계약과 가드레일을 가진 투자정보 에이전트를
  실제 사용성 과제로 평가했다"**로 잡는 것이 더 강하다.

## 6. 먼저 확정해야 할 결정

| 질문 | 왜 중요한가 |
|---|---|
| 에이전트는 여전히 **추천-free**인가? | 자본시장법 리스크 + 논문 1과의 일관성 |
| **동일 출판 시스템** 평가 vs 수정본 | 재현성·divergence 관리 |
| formative vs summative | 지표·표본·통계·IRB 범위 |
| 비교 조건(harness vs prompt-only, 출처 유·무) | 설계·counterbalancing·교란 통제 |
| 피험자 프로파일(리테일? 한국어 금융 리터러시) | 외적 타당도 |

## 7. 종합 판단

Throne은 우리 아이디어의 **상업적 검증**이자, 우리가 나아갈 방향의 참조점이다. 다만 목표는
복제가 아니라 **연구적으로 검증 가능한 투자정보 리서치 에이전트**다. 벤치마킹의 올바른
대상은 데이터 해자가 아니라 **워크플로·문서화·인용표시 UX**이며, 논문의 승부처는 Throne이
검증 가능하게 보여주지 못하는 **"코드로 강제된 안전성 → 사용자 신뢰"의 인과 연결**이다.
바로 다음 작업은 §5의 4개 산출물(경쟁제품 매트릭스, 안전 정책, MVP 화면 흐름, 태스크 5개)을
먼저 문서화하는 것이며, 구현은 그다음이다.
